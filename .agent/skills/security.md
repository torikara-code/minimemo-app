# セキュリティポリシー（Minimemo）

Minimemoはデスクトップ常駐型アプリとして、OSネイティブの特権とWebviewの境界を安全に維持する必要があります。

## 1. 原則
- **Zero Trust**: Webview（フロントエンド）からの入力はすべて信頼せず、Rust（バックエンド）側で検証を行う。
- **Fail Safely**: エラー時は内部情報を微塵もさらさない。ユーザー体験を壊さず、静かに安全な状態にフォールバックする。
- **Minimal Privilege**: 必要な権限（ファイルアクセス、クリップボード等）のみを Tauri 側で許可する。

## 2. IPC (Tauri Command) セキュリティ
フロントエンドから Rust 関数を呼び出す際は、以下の対策を講じること。
- **入力バリデーション**: 文字列の長さ、文字種、データ構造を Rust 側で厳密にチェックせよ。
- **許可パス制限**: `tauri-plugin-store` 等で使用する保存先は、アプリのデータディレクトリ内のみに限定し、任意のシステムファイルへのアクセスを遮断する。

## 3. XSS (クロスサイトスクリプティング) 対策
Markdown ハイライト層の実装において、ユーザーの入力を DOM に反映する際は細心の注意を払え。
- **エスケープ**: HTML特殊文字をエスケープした後に、Markdown の装飾タグ（`<b>`, `<i>` 等）を付加せよ。
- **innerHTML の制限**: `innerHTML` を直接操作するより、SolidJS の `createMemo` とテンプレートリテラル、または `innerComponent` を活用し、安全にレンダリングせよ。

## 4. ローカルデータと機密情報の保護
- **Secrets 管理**: APIキー、トライアル検証キー等の機密情報は、ソースコードにハードコードせず、ビルド時の環境変数（`.env` 等）から Rust 側で取得せよ。
- **ログの機密性**: ユーザーが入力したテキスト、パスワード、個人識別情報（PII）は絶対にログファイルに出力してはならない。

## 5. 安全なエラーハンドリング
- **詳細の保護**: ブラウザのコンソールやユーザー向けのトーストに、Rust 側のパニック内容や関数のスタックトレースを表示させてはならない。
- **ログへの隠蔽**: 詳細な不具合情報はバックエンドの `app.log` にのみ記録し、外部への流出を防ぐ。

## 6. 依存関係の健全性
- **pnpm audit**: 定期的に フロントエンドの `npm audit` および Rust の `cargo audit` を実行し、脆弱性のあるライブラリを排除せよ。
- **Tauri v2**: 公式のセキュリティアップデートを速やかに適用し、OSネイティブの脆弱性を放置しない。

## 7. AI 動作指針
1. ユーザー入力を扱う処理を書く際は、まず「この入力はスクリプトを含み得るか？」を自問し、適切なサニタイズを最初に行え。
2. OSコマンドの実行（`Command::new` 等）が必要な場合は、引数のエスケープとバリデーションを徹底せよ。
3. コードを書く際は便利さよりも安全なパターン（例：`unwrap()` ではなく `expect()` や `Result` 返却）を優先せよ。
